# ARCHITECTURE.md

Detailed architecture, data-schema and CSS/layout reference for this repo. Read this **before** touching the rendering logic in [script.js](script.js) / [widgets/](widgets/), the data schema in [exercises.json](exercises.json), or anything in [style.css](style.css).

> **Script layout.** `template.html` (root) is the **single renderer for every sheet** — the per-sheet `matura/<id>/index.html` copies are gone. At the bottom it loads a run of external classic scripts in order: `widgets/_helpers.js` (shared `wg*` helpers), the nine `widgets/*.js` widget files, `widgets/_registry.js` (the `WIDZETY` registry) **then** [script.js](script.js) (app logic). Order matters — `loadExercises` in `script.js` reads the `WIDZETY` const declared in `widgets/_registry.js`, which in turn reads the `widget*` functions declared in the individual widget files, and classic scripts share one global scope, so each declaring file must run before whatever reads it.

> **Which sheet is rendered** is chosen by the URL query param `?arkusz=<id>`, where `<id>` is a folder name under `matura/` (e.g. `template.html?arkusz=2026-maj`). `script.js` reads it into `SHEET_ID` (`new URLSearchParams(location.search).get("arkusz")` — **no default fallback**: a missing, empty, or unknown `?arkusz=` shows a "Błędny link" message with a link back to `index.html`, see "Data loading") and uses that one id for everything sheet-specific: the `fetch` path (`matura/<id>/exercises.json`), the localStorage key suffixes, and `mediaPath()` (see "Asset folder convention"). The landing page ([index.html](index.html)) links to each sheet as `template.html?arkusz=<id>`. There is no backend/redirect — it is a pure static page.

> **Keep this file in sync.** Whenever you change the code or styles it describes, update the matching section here in the same edit. Treat it as the source of truth for "how is this built and why".

## Rendering model

**Math rendering — KaTeX (offline).** KaTeX 0.16.11 is vendored in `vendor/katex/` (`katex.min.css` + `katex.min.js` + `contrib/auto-render.min.js` + woff2 fonts + LICENSE) and loaded synchronously in the `<head>`, so the page needs no internet for math. All math in `exercises.json` is written as `\( ... \)` (inline) / `\[ ... \]` (display, centred; multi-line via `\begin{aligned}` / `\begin{cases}`) — in JSON strings (same escaping as the old JS strings) that means `"\\( ... \\)"` and the aligned row separator is `"\\\\"`. The helper `renderMath(element)` (top of `script.js`) wraps `renderMathInElement` with those delimiters and `throwOnError: false`, guarded so a missing vendor file degrades to raw text instead of breaking the page. It is called in exactly two places: on the whole `exerciseClone` after it is filled (covers question, answers, hints, PF statements, fillIn labels, solution text/more) and on each step's freshly built DOM inside `showStep()` (step captions). Conventions: Polish decimal comma `6{,}38`, tangent `\operatorname{tg}`, CKE intervals `\langle -1, 7\rangle`, emphasised results `\boldsymbol{...}`. Formula *images* for zad 2, 6, 7, 8 and 10's function definition were replaced by KaTeX; the `<img>`s that remain in questions are genuine figures (graphs/drawings). `.katex` is sized `1.08em` in style.css so formulas blend with the 18px body text.

**Data loading.** Each sheet's data lives in `matura/<id>/exercises.json` (pure JSON — no comments, no functions; this file is the schema's documentation). The file is an **object** `{ "meta": {…}, "exercises": [ … ] }` — `meta` holds the per-sheet chrome (see "Sheet metadata" below), `exercises` is the array of exercise objects. `startSheet()` (bottom of `script.js`) `fetch`es `matura/${SHEET_ID}/exercises.json`, assigns `dane.exercises` to the global `exercises` array, calls `applySheetMeta(dane.meta)` to fill in the page's title/description/bar-title/PDF paths, and only then calls `loadExercises()`. `startSheet()` distinguishes the failure modes into distinct `.blad-wczytywania` messages instead of rendering nothing: a missing/empty `?arkusz=` (checked before any fetch) or a `fetch` that responds `!ok` (e.g. 404 — no such sheet folder) both show a **"Błędny link"** message with a link back to `index.html` (`pokazBladLinku()`); a `fetch` that throws (no response — `file://` blocks it, or the network is down) shows the "serve the directory with e.g. `npx serve`" hint; a broken JSON body shows a "file may be corrupted" note. `fetch` does not work over `file://`, which is why opening `template.html` directly lands in the second case.

`loadExercises()` (defined in [script.js](script.js)) runs once on page load. For each entry in the `exercises` array it clones the hidden `#exercise-template` node, fills in the fields, and appends the clone to `#exercises-wrapper`. All per-exercise behavior (answer click handling, hint/solution toggling, step-by-step nav, scoring) is wired up inside this same function at render time — there are no persistent component classes. The answers area dispatches on the exercise **type** (see schema below): `ABCD` single-choice buttons, `PF` true/false rows, `multiSelect` toggle buttons with partial credit, `fillIn` free-text inputs graded by a "Sprawdź" button (each blank worth an equal share of `maxScore`; user input and the `accepted` variants are both run through `normalizeAnswer`, which strips spaces and unifies interval brackets `⟨ ⟩ < >` → `[ ]`, typographic minus → `-`, `;` → `,`), `open` + `selfScore: true` a self-assessment button row (0..maxScore pkt). A per-exercise `setScore(points)` closure centralises scoring: it sets `earnedScore`, updates the badge text + colour tier (`high` = full, `mid` = partial, `low` = 0) and recomputes the total. The hint button hides itself when `hint` is empty (same pattern as the solution button).

**Progress persistence:** answers are saved to `localStorage` under a per-sheet key `matematykazen-postep-<SHEET_ID>` (e.g. `matematykazen-postep-2024-grudzien` — the id is the `?arkusz=` value / folder name, so sheets never share progress) as *what the user clicked/typed* (`{abcd, pf, multi, self, fill}` per exercise). On load, `loadExercises` restores by **replaying those clicks** through the normal handlers, so scoring/colours use the exact same code path — there is no second grading implementation. A save whose exercise count differs from `exercises.length` is discarded (schema changed). The save is wiped (and the page reloaded) by `#reset-scores` in the `#bar-menu` dropdown — the only such button; a second one under the last exercise (`#clear-progress`) was removed as a duplicate. Multi-part exercises (Zadanie 12, 17) have a separate "parent" entry holding only the shared intro (`maxScore: 0`) — for those the score badge is `remove()`d outright (hiding it wouldn't survive the score-view switcher, which re-sets `display` on all `.exercise-score`).

The widget layer lives in [widgets/](widgets/) (all loaded before `script.js`): `widgets/_helpers.js` holds the shared helpers (`wgElement`, `wgCanvas`, `wgPointerPos`, `wgDraggable` for pointer-event dragging, `wgPrzyciagnij` soft-snapping of dragged values to a list of target values — used by widgets 1/9/12.1/18 so the "task situation" is easy to hit exactly, `wgStrzalka` arrows, `WG_KOLORY` palette; **widget math is KaTeX too**: `wgMath(tex)` is a *memoized* `katex.renderToString` (never the auto-render — live readouts update dozens of times per second while dragging; widget values are quantised so the cache stays small), `wgTexLiczba(v, maxFrac, minFrac)` formats a number for TeX Polish-style (`{,}` decimal comma, `\,` thousand gaps), and `wgUstawHTML(el, html)` swaps a readout's innerHTML only when the string actually changed — titles and the zad 10 formula list render once at build time, readouts go through `wgUstawHTML`; inputs can't live inside a KaTeX formula, so zad 1's `|x + ▢| = ▢` control row is KaTeX fragments glued around the two `<input>`s). Each interactive solution is one global `widget*` function in its own file — `widgets/osLiczbowa.js` → `widgetOsLiczbowa` (zad 1), `widgets/procentSkladany.js` → `widgetProcentSkladany` (5), `widgets/nierownoscKwadratowa.js` → `widgetNierownoscKwadratowa` (9), `widgets/funkcjaPrzedzialami.js` → `widgetFunkcjaPrzedzialami` (10), `widgets/parabola.js` → `widgetParabola` (12.1), `widgets/ciagArytmetyczny.js` → `widgetCiagArytmetyczny` (15), `widgets/koloTrygonometryczne.js` → `widgetKoloTrygonometryczne` (18), `widgets/katWpisany.js` → `widgetKatWpisany` (20), `widgets/prostopadloscian.js` → `widgetProstopadloscian` (30). Widgets build all their DOM inside the container they receive (no fixed ids; every lookup is scoped), so they can coexist on one page. `exercises.json` references them **by name**: `"solutionWidget": "widgetX"` (JSON can't hold functions). The mapping name → function is the `WIDZETY` registry in `widgets/_registry.js`, the last widget-related `<script>` tag in template.html (so every `widget*` function is already a defined global by the time it runs); `loadExercises` (in `script.js`) looks the widget up there (unknown names `console.warn` and render nothing). Adding a widget = new file in `widgets/` with a `widget*` function + an entry in `WIDZETY`.

## Exercise data schema (`exercises.json`)

### Sheet metadata (`meta`)

The top-level `meta` object carries everything `template.html` needs to know about *which* sheet it is rendering (so the HTML itself is sheet-agnostic — its `<title>`, `<meta name="description">`, `#exercises-sheet-title` and the zasady-oceniania `<object>` start empty/generic and are filled by `applySheetMeta()` at load). Fields:

- `pageTitle` — browser-tab `<title>`.
- `metaDescription` — `<meta name="description">` content.
- `sheetTitle` — text for the centred `#exercises-sheet-title` in the top bar.
- `zasadyPdf` — path to the CKE marking/answer-key PDF for the zasady-oceniania panel, **relative to the sheet folder** (`matura/<id>/`), e.g. `"matematyka-2024-grudzien-probna-podstawowa-odpowiedzi.pdf"`. It is resolved through `mediaPath()` (like images/videos), so the file lives alongside `exercises.json` in the sheet folder.
- `tablicaPdfDefaultPage` — page number the formula-sheet panel opens on by default (the formula sheet itself, `wybrane_wzory_matematyczne.pdf`, is shared and stays in the repo root — `TABLICE_PDF` in `script.js`, not per-sheet).

**Styling rule for `exercises.json`:** only put inline HTML/CSS (e.g. a `style="…"` on an `<img>`) into an exercise's data when it is genuinely specific to that one exercise — e.g. Zad 2's inline number image opting out of the block/centred `.question img` rule with `display:inline-block; vertical-align:middle`. Anything that should apply to *all* exercises (answer-button alignment, generic figure sizing, colours, spacing, …) belongs in [style.css](style.css), never copy-pasted per entry. If you catch yourself adding the same inline style to several exercises, move it to a CSS rule instead.

Each element of `exercises` is an object with this shape (multi-part exercises are separate array entries: a "parent" entry with the shared intro and `maxScore: 0`, then "Zadanie 12.1", "12.2", … as normal entries — the planned `number`/`numberSection` fields are still not implemented):

- `type` — `"ABCD"` | `"PF"` | `"multiSelect"` | `"fillIn"` | `"open"`. Optional: when absent it is inferred (`answers` non-empty → `ABCD`, empty → `open`), so plain multiple-choice entries usually omit it.
- `question` — HTML string, rendered into `.question`. Often embeds `<img>` tags pointing into the exercise's `media/zadN/` folder (sheet-relative; resolved by `mediaPath` — see "Asset folder convention").
- `answers` — array of HTML strings for choice buttons (`ABCD`/`multiSelect`). Empty array (`[]`) for `PF` and `open` exercises.
- `correctAnswerIndex` — (`ABCD`) 0-based index into `answers` of the correct option (`0` = A, `1` = B, …). `-1` or a missing field marks an unfilled exercise: clicking an answer just toggles a single `?` feedback marker instead of grading. `loadExercises` `console.warn`s if the index is out of range.
- `correctAnswerIndices` — (`multiSelect` only) array of correct indices; its length is the number of picks required. Clicks toggle a light-blue `.selected` state; once the required count is selected the picks are graded (green/red) and partial credit is granted (`maxScore · hits / required`, rounded). Deselecting re-opens the choice. When the pick-limit is already reached, clicking a *new* answer replaces the earliest selection (a sliding window — the `selected` `Set`'s insertion order makes the oldest pick its first element) instead of doing nothing, so the student never has to manually un-click to change an answer.
- `statements` — (`PF` only) array of `{ text, answer }` (`answer: true` = P). Each statement renders as a `.pf-row` with its own P/F buttons graded immediately; the exercise scores `maxScore` only when **all** statements are answered correctly, else 0 (CKE-style).
- `blanks` — (`fillIn` only, currently zad 10 and 29) array of `{ label, accepted }`: `label` is the sentence HTML rendered left of the text input, `accepted` the list of recognised answers (compared after `normalizeAnswer` on both sides, so one canonical form like `"(-4,4]"` covers `(−4, 4⟩`, `(-4;4>` etc.; list numeric variants like `"6,38"`/`"6.38"` explicitly). Editing an input clears its previous green/red mark; "Sprawdź" grades all blanks and scores `maxScore · hits / blanks` (rounded).
- `maxScore` — the exercise's point value. `0` marks a "parent" entry (shared intro of a multi-part exercise): no answers UI, no badge (removed from DOM), contributes nothing to the total. All answer paths go through the shared `setScore(points)` helper; `totalScore` is recomputed as the sum of every `earnedScore` (`updateTotalScore()`), so scoring is idempotent and self-correcting — no `isScoreGiven` flag or manual add/subtract.

**Immediate vs. deferred grading (closed exercises).** The "graded immediately / once the required count is selected" behavior above for `ABCD`/`PF`/`multiSelect` is the default (**ON**) of a global toggle — "pokazuj poprawność od razu", in the `#bar-menu` dropdown, stored in `localStorage` key `matematykazen-natychmiastowa-poprawnosc` (global, **no** sheet suffix — like the theme). When **OFF** ("sprawdź później"), clicking an answer only sets a neutral `.selected` state; a floating "sprawdź" button (`.answer-check-floating`, `position: absolute` to the right of `.answers-container`, so it never changes the exercise's height) then reveals the coloring/score and disappears. Each closed branch in `loadExercises` exposes an idempotent `ocen*()` that colors from the stored selection; the per-question button and the "sprawdź wszystkie odpowiedzi" action (in `#bar-menu` **and** the sheet footer, both only shown under `body.reczne-sprawdzanie` outside exam mode — they iterate the `oczekujaceSprawdzenia` registry) call it. `stan.sprawdzone` is persisted so a reload keeps revealed answers colored (with their score) while selected-but-unchecked ones return neutral with the button. **Exam mode always forces immediate** (`natychmiastowaOcenaAktywna()` = the pref **or** `body.tryb-egzaminu`): grading runs as usual under the hood, exam.css hides the colors, and no "sprawdź" buttons appear. `fillIn` is unaffected — it keeps its own "Sprawdź" button.
- `selfScore` — (`open` only) `true` renders the self-assessment row ("Rozwiąż na kartce, porównaj z rozwiązaniem i oceń się:" + buttons `0 pkt`..`maxScore pkt`) that implements the product's off-platform-solve/self-grade flow.
- `finalAnswer` — (`open` + `selfScore`, **optional**) `{ label, placeholder, accepted }`: renders a single **"ostateczna odpowiedź"** text input under the tok-rozwiązania textarea (`.final-answer-container`) with a "Sprawdź" button. On check the input is graded exactly like `fillIn` (both the input and every `accepted` variant run through `normalizeAnswer` in [app/answers.js](app/answers.js); note `normalizeAnswer` unifies `⟨⟩<>[]`→`[]` but leaves round parens literal, so a closed interval `⟨-1,7⟩`/`[-1,7]` matches while the open `(-1,7)` does not) and coloured green/red. It **does not touch `earnedScore`** — points still come from the self-score buttons; it is only the single *automatic* correctness signal open exercises get. **Deliberately lives OUTSIDE `.self-score-container`, so — unlike the checklist and solutions — it (and its colour) stay available DURING the exam** (`.final-answer-input` is intentionally left out of exam.css's colour-neutralisation). The entered value + revealed colour persist under `stan.koncowa`. Absent → no final-answer field.
- `gradingCriteria` — (`open` + `selfScore`, **optional**) array of HTML strings (KaTeX allowed): renders a **helper checklist** (`.grading-criteria`, checkboxes) at the top of `.self-score-container` — each item is an element of the official CKE marking key ("Zaznacz, co znalazło się w Twoim rozwiązaniu…"). It is **purely advisory: it does NOT sum points automatically** — the student still picks the final points with the `0 pkt`..`maxScore pkt` buttons. Checkbox state persists under `stan.kryteria` (array of bools). Being inside `.self-score-container`, it is hidden during the exam like the rest of the self-assessment UI. Currently wired as a prototype on Zad 8 and Zad 9 of `2024-grudzien` (per issues/formularz-oceniania-otwarte.md); other open exercises get criteria added to `exercises.json` later. A single general legal disclaimer (`.samoocena-disclaimer`, sheet footer in template.html) covers both features once — the site is not an examiner.
- `hint` — HTML shown/hidden by the "Podpowiedź" button; empty string hides the button entirely.
- `formulasPage` — page number into `wybrane_wzory_matematyczne.pdf`, or `null` to hide the "Pokaż potrzebne wzory" button. Clicking it calls `openFormulasAtPage(page)`, which replaces the PDF `<object>` inside `#tablica-wzorow-panel` entirely (changing just `data` doesn't reliably reload PDF viewers) and then shows the panel via `showFormulasPanel()`. Show/hide is centralised in `showFormulasPanel()`/`hideFormulasPanel()` (they toggle the panel's `display` and keep the `#toggle-tablica` label in sync); the bar button and the in-panel close button (`#tablica-close`, the ✕ in its top-right corner) both go through them. A mirrored second panel `#zasady-oceniania-panel` (left-hand side) shows the official CKE answer/marking PDF whose path comes from `meta.zasadyPdf` (set on the `<object id="zasady-oceniania">` by `applySheetMeta`, resolved through `mediaPath` + `encodeURI`); it has its own `#toggle-zasady` bar button and `showGradingRules()`/`hideGradingRules()`/`#zasady-close`, but no per-page targeting. Both panels are "unlocked": `makePanelDraggable(panel)` makes the top strip (`.panel-uchwyt`) drag the panel (switching it to `left/top` positioning) and the bottom-right grip (`.panel-rozmiar`) resize it, using pointer capture on the handles because the PDF `<object>` swallows mouse events. The **only** drag constraint is on `top`: it is clamped to the `#top-bar`'s bottom edge (`getBoundingClientRect().bottom`, read fresh on each `pointerdown`) so the drag strip can never slide under the fixed top bar and become ungrabbable; `left` (and hence right/bottom) is deliberately **unclamped** — the panel may be dragged partly or fully off any other screen edge. Resize is unconstrained.
- `solutionText` / `solutionTextMore` — HTML for the solution panel; `solutionTextMore` is behind an extra "Pokaż więcej" toggle and its container auto-hides when empty.
- `solutionStepByStep` — either `null` or an array of `{ type: "video"|"image"|"text", src, text }` steps rendered one at a time with prev/next navigation inside the solution panel. **Step switching is double-buffered against flicker:** `showStep()` builds the next step's DOM detached (and KaTeX-renders it), and when the *previous* step is still on screen and the new step has a video, the old step stays visible (frozen on its last frame) until the new video has decoded its first frame (`loadeddata`, `readyState >= 2` shortcut for cached files, 1.5 s `setTimeout` fallback) — only then does `replaceChildren` swap the content and `video.play()` start playback (the `autoplay` attribute is stripped so the buffered video doesn't start early). A per-exercise `stepSwapToken` plus a `krokWstawiony` flag guard against rapid ◄/► clicking and against the `loadeddata`+timeout pair both firing. The nav (counter, disabled arrows, `markCorrectAnswer` on the last step) updates immediately on click, not at swap time. `renderStep()` still returns a plain HTML string (video steps are wrapped in a `.step-video` div holding the `<video>`, a small `.video-overlay-icon` marker, horizontally centred at the bottom of the film — two vertical bars (pause) drawn with `::before`/`::after` when the video is paused mid-play, or a ↺ replay arrow once it has ended (driven by `.paused`/`.ended` classes that `showStep` toggles from the video's `paused`/`ended` state; clicking an ended video replays it from the start) and a thin grey `.video-progress`/`.video-progress-bar` (the bar is driven by a `requestAnimationFrame` loop in `showStep` that runs only while the video plays and self-terminates when it pauses/ends or the step's `<video>` leaves the DOM (`!isConnected`); no CSS transition — an earlier `timeupdate`+transition combo made the bar move rubber-band-like); the caption goes in a `.step-comment` div); the wiring of the real `<video>` lives in `podepnijSterowanieWideo(video)`, called at swap time: it sets `defaultPlaybackRate`/`playbackRate` (currently `1`), and wires click-to-toggle play/pause, a `.paused` class on `.step-video` (which reveals the ▶ icon whenever the video is paused or ended), and the rAF progress-bar loop described above. Doing all this on the attached element is deliberate — JS properties/listeners can't be set on the throwaway string `renderStep` returns. Reaching the last step calls `markCorrectAnswer()`, which reveals the correct answer button (guarded against a missing button) — i.e. finishing the walkthrough is how open-style exercises "give away" the answer. The "Rozwiązanie" button itself is hidden in `loadExercises` when an exercise has no solution content at all (no text/steps/interactive/more).
- `solutionWidget` — either `null` or the **name** (string) of one of the `widget*` functions described under "Rendering model", e.g. `"widgetOsLiczbowa"`; `loadExercises` resolves it through the `WIDZETY` registry in `widgets/_registry.js` and calls the function with the exercise's `.solution-interactive-container`. Nine exercises currently have widgets (1, 5, 9, 10, 12.1, 15, 18, 20, 30).

## Asset folder convention

Every sheet keeps its assets under **`matura/<id>/media/zadN/`** (one `zadN/` folder per exercise number, not array index), e.g. `matura/2024-grudzien/media/zad1/zad1rozw_step1.mp4` (rozwiązanie = solution), `matura/2024-grudzien/media/zad10/zad10rys.png` (a real figure). In `exercises.json` these are written **relative to the sheet folder** with the `media/` prefix — `"media/zad1/zad1rozw_step1.mp4"`, `<img src="media/zad10/zad10rys.png">` — the same convention for both sheets.

**Path resolution — `mediaPath(src)`.** Because `template.html` renders from the repo root (not from inside the sheet folder), a bare `media/…` path would resolve against the root. `mediaPath(src)` (top of `script.js`) prepends `matura/${SHEET_ID}/` to any non-absolute path (leaves `http(s):`/`data:`/`/…` untouched). It is applied in three places: the `<img>`/`<video>`/`<source>` `src` inside step HTML (`renderStep`), the next-step preload, and — via a `querySelectorAll("img[src]")` sweep of the freshly built exercise clone (just before `renderMath`) — every `<img>` embedded in the question/hint/solution HTML. The `meta.zasadyPdf` path goes through `mediaPath` too. So **paths in `exercises.json` stay sheet-relative; `mediaPath` is the single place that joins them to the sheet folder** — don't hard-code `matura/<id>/` into the data.

Keep filenames **lowercase** — `zad17.PNG` once broke on case-sensitive hosting and had to be renamed. Since the KaTeX migration (2026-07-06) the formula crops `zad2/zad2.png`, `zad6/zad6*.png`, `zad7/zad7*.png`, `zad8/zad8.png`, `zad10/zad10.png` and the whole `zad16/` folder were no longer referenced (their content is typeset in KaTeX) and were deleted. The 2026-maj figures (`media/zad12/…` etc.) are still to be cut from that sheet's exam PDF, so those paths currently 404 — expected until the images are added.

Each sheet's own exam + answer-key PDFs live in its folder too (`matura/<id>/matematyka-<id>-…-odpowiedzi.pdf`, referenced by `meta.zasadyPdf`; the plain exam PDF is archived alongside). The shared formula sheet `wybrane_wzory_matematyczne.pdf` stays in the repo root. Other/unwired sheets and their text extracts sit in `inne arkusze PDF/`.

Step-by-step solution videos are produced externally with **Manim** (the Python math-animation library), then exported as short `.mp4` files and dropped into the relevant `media/zadN/` folder — they aren't generated or editable from within this repo, only referenced and played back (see `solutionStepByStep` above).

## Scoring / UI chrome

`maxTotalScore` is computed inside `loadExercises()` as the sum of all `maxScore` fields (the old hand-maintained constant at the top of the data file is gone — nothing to keep in sync). It renders the running total in `#total-score`. `#score-switch-button` (in the `#bar-menu` dropdown) cycles through three display modes (all per-exercise scores / total only / nothing) by toggling `.exercise-score` element visibility, purely client-side, no persistence. `#show-all-solutions` (also in `#bar-menu`) is a toggle that opens/closes every exercise's solution panel by clicking each exercise's own "Rozwiązanie" button (collected in the `wszystkieRozwiazania` array during render), skipping exercises already in the target state; its label flips between "pokaż/schowaj wszystkie rozwiązania". `#reset-scores` (third item in `#bar-menu`) clears the saved progress and reloads. Answer progress itself *is* persisted — see "Progress persistence" under Rendering model.

## Exam mode (tryb egzaminacyjny)

"Rozpocznij próbny egzamin" (`#egzamin-start`, in the `#bar-menu` dropdown) runs a timed mock exam. **Design principle: grading and progress-saving keep running through the exact same code path as normal mode** — the mode only adds a `body.tryb-egzaminu` class, and style.css under that class hides everything that reveals the result: `.correct`/`.incorrect` are restyled to the neutral blue "selected" look, score badges / `#total-score` / hint / solution / self-score rows / `#toggle-zasady` (the CKE answer-key PDF) / the other `#bar-menu` items are `display: none !important` (important because some of those get inline `display` from JS). The CKE formula sheet (`#toggle-tablica`, per-exercise "Pokaż potrzebne wzory") deliberately stays available — it is allowed on the real matura.

Mechanics (all in [app/exam.js](app/exam.js)): state is `{ start: timestamp }` under its own per-sheet localStorage key `matematykazen-egzamin-<SHEET_ID>` (`KLUCZ_EGZAMINU`, e.g. `…-2024-grudzien`), fully independent of the answer-progress key, so a page refresh resumes the exam (the class is applied at script start, before any rendering — no flash of scores). Start/end each have two buttons wired to one shared handler (`#egzamin-start` in `#bar-menu` + `#egzamin-start-stopka` in the sheet footer both call `startExamPrompt()`; `#egzamin-koniec` — a static button placed *after* `#exercises-wrapper`, `display: none` outside exam mode — + `#egzamin-koniec-bar` in the top bar both call `finishExamPrompt()`), so start/end behavior can't drift between the two copies. Starting the exam `confirm()`s, wipes the saved progress and the `KLUCZ_OCENIANIA` self-assessment-phase flag (an exam starts on a clean sheet) and reloads. `tickExam()` ticks every second into `#egzamin-timer` (tabular-nums; `.malo-czasu` turns it red under 10 min) and auto-ends the exam at 0 — guarded on `exercises.length` so that a timer that expired while the tab was closed ends the exam right after the data loads (`startSheet` calls one tick after `loadExercises`). Ending (either button, or timeout) removes the key + class and fills the `#egzamin-podsumowanie` overlay: closed-exercise score (`earnedScore` sum vs non-`selfScore` `maxScore` sum), percentage, time used, and an explicit note that **open (selfScore) exercises are scored by self-assessment only now, after the exam** — their rows and the solutions just got unlocked, and self-graded points flow into the same total. Answers given during the exam keep their colours-revealed state after it ends (same DOM, class removed). A permanent `#exercises-mode-subtitle` (small grey line under the sheet title in `#bar-center`) always shows the current mode — `updateModeSubtitle()` sets it to `"tryb egzaminu"` when `body.tryb-egzaminu` is present (called from `enableExamMode()`) and back to `"tryb ćwiczenia"` otherwise (called from `finishExam()`); it is not exam-only chrome. It is hidden under 900px alongside `#exercises-sheet-title`.

Ending the exam also flips on the **"oceń się" phase** (`ustawFazeOceniania(true)` in `finishExam()`, [app/indicators.js](app/indicators.js)): a `KLUCZ_OCENIANIA` localStorage flag (own per-sheet key, survives refresh) that, while set, shows a floating yellow dot (`.wskaznik-otwarte`) next to every filled-in but not-yet-self-graded open exercise — click scrolls to it, and grading it (or manually dismissing via the phase's own hide control) clears its dot; the phase auto-turns off once every open exercise is graded. Two tabs of the same sheet do **not** stay in sync while an exam is running — `finishExam()`/`KLUCZ_EGZAMINU` removal only fires in the tab that clicked end/timed out; other tabs of the same sheet learn about it via a `storage` event listener (fires only in *other* tabs) that shows a "finished in another tab" overlay prompting a reload, rather than trying to render a second, possibly-inconsistent summary from that tab's own in-memory scores.

## Zgłaszanie błędów w zadaniu (report an error)

Lives in [app/report.js](app/report.js) (loaded just before `render.js`, because `loadExercises` calls into it per exercise). Under every rendered exercise it appends a discreet text button `.report-error-link` ("zgłoś błąd w tym zadaniu") via `dodajLinkZgloszenia(exerciseClone)` — the exercise number is parsed from the question text (`/Zadanie (\d+)/`, same approach as the "oceń się" indicators, because array index ≠ CKE numbering). Clicking it opens a single shared modal (`#zglos-blad-overlay` / `.zglos-blad-okno`, static in template.html, styled like `#egzamin-podsumowanie`; `display: flex` toggled by JS), filled at open time with the exercise context and a live JSON preview of the auto-attached data.

**Toggle:** `#zglos-blad-toggle` in the `#bar-menu` dropdown enables/disables the link **globally** (localStorage key `matematykazen-zglaszanie-bledow`, no sheet suffix — like the theme; only `"0"` disables, default on), by toggling `body.bez-zglaszania` which the CSS uses to hide all `.report-error-link`. Same pattern as the other menu toggles (`#natychmiastowa-toggle`, `#wskazniki-tryb-toggle`).

**Submission** goes to **Formspree** via `fetch` (AJAX, no page reload). `FORMSPREE_ENDPOINT` at the top of `report.js` is a **placeholder** (`"TODO-WKLEJ-ENDPOINT-FORMSPREE"`) — Henrich must create a Formspree form and paste its `https://formspree.io/f/xxxxxxxx` URL there before it works in production; until then the submit shows a clear "Zgłoszenia nie są jeszcze skonfigurowane" toast (guarded by a regex on the endpoint) instead of a confusing network error. All form fields are **optional** (description textarea + email, both explicitly labelled "(opcjonalnie)") — an empty submit with just the auto-data is valid and useful. **Auto-attached (user types none of it):** exercise number, `SHEET_ID`, `location.href`, effective theme (`aktualnyMotyw()` distinguishes manual jasny/ciemny from auto and resolves what auto currently means), exam-vs-practice mode, `navigator.userAgent`, and the whole of `localStorage` (`JSON.stringify` of every key). A `<details>` in the modal previews this for transparency.

**Anti-spam:** a honeypot field `input[name="_gotcha"]` (Formspree's native honeypot name), hidden from humans by CSS (`.zglos-blad-hp` positioned off-screen, **not** `display:none`, so bots still fill it) — if filled, the submit fakes a success toast and sends nothing. Plus light **throttling**: no send more than once per minute (`matematykazen-zglos-ostatnia` timestamp, global) to protect the Formspree free tier's 50-submissions/month cap. The throttle timestamp is set only when the request actually **reaches** the server (any HTTP response), not on a pure network failure (so a user can retry immediately after losing connection) and not for the placeholder short-circuit.

**Feedback:** a toast (`.zglos-toast`, created lazily, bottom-centre, green `--correct` border for success / red `--incorrect` for errors) — "Dziękujemy, zgłoszenie wysłane." on `res.ok`; distinct error messages for a non-ok response (e.g. 429 = monthly limit), a network exception, throttle, and the unconfigured placeholder — never a silent failure.
