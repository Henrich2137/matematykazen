# ARCHITECTURE.md

Detailed architecture, data-schema and CSS/layout reference for this repo. Read this **before** touching the rendering logic in [script.js](script.js) / [solutionsInteractive.js](solutionsInteractive.js), the data schema in [exercises.json](exercises.json), or anything in [style.css](style.css).

> **Script layout.** `template.html` (root) is the **single renderer for every sheet** ‚Äî the per-sheet `matura/<id>/index.html` copies are gone. At the bottom it loads two external classic scripts in order: [solutionsInteractive.js](solutionsInteractive.js) (widgets + `WIDZETY` registry, back in the root) **then** [script.js](script.js) (app logic). Order matters ‚Äî `loadExercises` in `script.js` reads the `WIDZETY` const declared in `solutionsInteractive.js`, and classic scripts share one global scope, so the declaring file must run first.

> **Which sheet is rendered** is chosen by the URL query param `?arkusz=<id>`, where `<id>` is a folder name under `matura/` (e.g. `template.html?arkusz=2026-maj`); no param falls back to `2024-grudzien`. `script.js` reads it into `SHEET_ID` (`new URLSearchParams(location.search).get("arkusz")`) and uses that one id for everything sheet-specific: the `fetch` path (`matura/<id>/exercises.json`), the localStorage key suffixes, and `mediaPath()` (see "Asset folder convention"). The landing page ([index.html](index.html)) links to each sheet as `template.html?arkusz=<id>`. There is no backend/redirect ‚Äî it is a pure static page.

> **Keep this file in sync.** Whenever you change the code or styles it describes, update the matching section here in the same edit. Treat it as the source of truth for "how is this built and why".

## Rendering model

**Math rendering ‚Äî KaTeX (offline).** KaTeX 0.16.11 is vendored in `vendor/katex/` (`katex.min.css` + `katex.min.js` + `contrib/auto-render.min.js` + woff2 fonts + LICENSE) and loaded synchronously in the `<head>`, so the page needs no internet for math. All math in `exercises.json` is written as `\( ... \)` (inline) / `\[ ... \]` (display, centred; multi-line via `\begin{aligned}` / `\begin{cases}`) ‚Äî in JSON strings (same escaping as the old JS strings) that means `"\\( ... \\)"` and the aligned row separator is `"\\\\"`. The helper `renderMath(element)` (top of `script.js`) wraps `renderMathInElement` with those delimiters and `throwOnError: false`, guarded so a missing vendor file degrades to raw text instead of breaking the page. It is called in exactly two places: on the whole `exerciseClone` after it is filled (covers question, answers, hints, PF statements, fillIn labels, solution text/more) and on each step's freshly built DOM inside `showStep()` (step captions). Conventions: Polish decimal comma `6{,}38`, tangent `\operatorname{tg}`, CKE intervals `\langle -1, 7\rangle`, emphasised results `\boldsymbol{...}`. Formula *images* for zad 2, 6, 7, 8 and 10's function definition were replaced by KaTeX; the `<img>`s that remain in questions are genuine figures (graphs/drawings). `.katex` is sized `1.08em` in style.css so formulas blend with the 18px body text.

**Data loading.** Each sheet's data lives in `matura/<id>/exercises.json` (pure JSON ‚Äî no comments, no functions; this file is the schema's documentation). The file is an **object** `{ "meta": {‚Ä¶}, "exercises": [ ‚Ä¶ ] }` ‚Äî `meta` holds the per-sheet chrome (see "Sheet metadata" below), `exercises` is the array of exercise objects. `startSheet()` (bottom of `script.js`) `fetch`es `matura/${SHEET_ID}/exercises.json`, assigns `dane.exercises` to the global `exercises` array, calls `applySheetMeta(dane.meta)` to fill in the page's title/description/bar-title/PDF paths, and only then calls `loadExercises()`. `fetch` does not work over `file://` ‚Äî in that case (or any other load failure) the page shows a `.blad-wczytywania` message telling the user to serve the directory (e.g. `npx serve`) instead of rendering nothing.

`loadExercises()` (defined in [script.js](script.js)) runs once on page load. For each entry in the `exercises` array it clones the hidden `#exercise-template` node, fills in the fields, and appends the clone to `#exercises-wrapper`. All per-exercise behavior (answer click handling, hint/solution toggling, step-by-step nav, scoring) is wired up inside this same function at render time ‚Äî there are no persistent component classes. The answers area dispatches on the exercise **type** (see schema below): `ABCD` single-choice buttons, `PF` true/false rows, `multiSelect` toggle buttons with partial credit, `fillIn` free-text inputs graded by a "Sprawd≈∫" button (each blank worth an equal share of `maxScore`; user input and the `accepted` variants are both run through `normalizeAnswer`, which strips spaces and unifies interval brackets `‚ü® ‚ü© < >` ‚Üí `[ ]`, typographic minus ‚Üí `-`, `;` ‚Üí `,`), `open` + `selfScore: true` a self-assessment button row (0..maxScore pkt). A per-exercise `setScore(points)` closure centralises scoring: it sets `earnedScore`, updates the badge text + colour tier (`high` = full, `mid` = partial, `low` = 0) and recomputes the total. The hint button hides itself when `hint` is empty (same pattern as the solution button).

**Progress persistence:** answers are saved to `localStorage` under a per-sheet key `matematykazen-postep-<SHEET_ID>` (e.g. `matematykazen-postep-2024-grudzien` ‚Äî the id is the `?arkusz=` value / folder name, so sheets never share progress) as *what the user clicked/typed* (`{abcd, pf, multi, self, fill}` per exercise). On load, `loadExercises` restores by **replaying those clicks** through the normal handlers, so scoring/colours use the exact same code path ‚Äî there is no second grading implementation. A save whose exercise count differs from `exercises.length` is discarded (schema changed). The save is wiped (and the page reloaded) by `#reset-scores` in the `#bar-menu` dropdown ‚Äî the only such button; a second one under the last exercise (`#clear-progress`) was removed as a duplicate. Multi-part exercises (Zadanie 12, 17) have a separate "parent" entry holding only the shared intro (`maxScore: 0`) ‚Äî for those the score badge is `remove()`d outright (hiding it wouldn't survive the score-view switcher, which re-sets `display` on all `.exercise-score`).

The widget layer lives in [solutionsInteractive.js](solutionsInteractive.js) (loaded before `script.js`), a `WID≈ªETY INTERAKTYWNE` section: shared helpers (`wgElement`, `wgCanvas`, `wgPointerPos`, `wgDraggable` for pointer-event dragging, `wgPrzyciagnij` soft-snapping of dragged values to a list of target values ‚Äî used by widgets 1/9/12.1/18 so the "task situation" is easy to hit exactly, `wgStrzalka` arrows, `WG_KOLORY` palette; **widget math is KaTeX too**: `wgMath(tex)` is a *memoized* `katex.renderToString` (never the auto-render ‚Äî live readouts update dozens of times per second while dragging; widget values are quantised so the cache stays small), `wgTexLiczba(v, maxFrac, minFrac)` formats a number for TeX Polish-style (`{,}` decimal comma, `\,` thousand gaps), and `wgUstawHTML(el, html)` swaps a readout's innerHTML only when the string actually changed ‚Äî titles and the zad 10 formula list render once at build time, readouts go through `wgUstawHTML`; inputs can't live inside a KaTeX formula, so zad 1's `|x + ‚ñ¢| = ‚ñ¢` control row is KaTeX fragments glued around the two `<input>`s) plus one global `widget*` function per interactive solution ‚Äî `widgetOsLiczbowa` (zad 1), `widgetProcentSkladany` (5), `widgetNierownoscKwadratowa` (9), `widgetFunkcjaPrzedzialami` (10), `widgetParabola` (12.1), `widgetCiagArytmetyczny` (15), `widgetKoloTrygonometryczne` (18), `widgetKatWpisany` (20), `widgetProstopadloscian` (30). Widgets build all their DOM inside the container they receive (no fixed ids; every lookup is scoped), so they can coexist on one page. `exercises.json` references them **by name**: `"solutionInteractive": "widgetX"` (JSON can't hold functions). The mapping name ‚Üí function is the `WIDZETY` registry at the end of `solutionsInteractive.js`; `loadExercises` (in `script.js`) looks the widget up there (unknown names `console.warn` and render nothing). Adding a widget = new `widget*` function + an entry in `WIDZETY`.

## Exercise data schema (`exercises.json`)

### Sheet metadata (`meta`)

The top-level `meta` object carries everything `template.html` needs to know about *which* sheet it is rendering (so the HTML itself is sheet-agnostic ‚Äî its `<title>`, `<meta name="description">`, `#exercises-sheet-title` and the zasady-oceniania `<object>` start empty/generic and are filled by `applySheetMeta()` at load). Fields:

- `pageTitle` ‚Äî browser-tab `<title>`.
- `metaDescription` ‚Äî `<meta name="description">` content.
- `sheetTitle` ‚Äî text for the centred `#exercises-sheet-title` in the top bar.
- `zasadyPdf` ‚Äî path to the CKE marking/answer-key PDF for the zasady-oceniania panel, **relative to the sheet folder** (`matura/<id>/`), e.g. `"matematyka-2024-grudzien-probna-podstawowa-odpowiedzi.pdf"`. It is resolved through `mediaPath()` (like images/videos), so the file lives alongside `exercises.json` in the sheet folder.
- `tablicaPdfDefaultPage` ‚Äî page number the formula-sheet panel opens on by default (the formula sheet itself, `wybrane_wzory_matematyczne.pdf`, is shared and stays in the repo root ‚Äî `TABLICE_PDF` in `script.js`, not per-sheet).

**Styling rule for `exercises.json`:** only put inline HTML/CSS (e.g. a `style="‚Ä¶"` on an `<img>`) into an exercise's data when it is genuinely specific to that one exercise ‚Äî e.g. Zad 2's inline number image opting out of the block/centred `.question img` rule with `display:inline-block; vertical-align:middle`. Anything that should apply to *all* exercises (answer-button alignment, generic figure sizing, colours, spacing, ‚Ä¶) belongs in [style.css](style.css), never copy-pasted per entry. If you catch yourself adding the same inline style to several exercises, move it to a CSS rule instead.

Each element of `exercises` is an object with this shape (multi-part exercises are separate array entries: a "parent" entry with the shared intro and `maxScore: 0`, then "Zadanie 12.1", "12.2", ‚Ä¶ as normal entries ‚Äî the planned `number`/`numberSection` fields are still not implemented):

- `type` ‚Äî `"ABCD"` | `"PF"` | `"multiSelect"` | `"fillIn"` | `"open"`. Optional: when absent it is inferred (`answers` non-empty ‚Üí `ABCD`, empty ‚Üí `open`), so plain multiple-choice entries usually omit it.
- `question` ‚Äî HTML string, rendered into `.question`. Often embeds `<img>` tags pointing into the exercise's `media/zadN/` folder (sheet-relative; resolved by `mediaPath` ‚Äî see "Asset folder convention").
- `answers` ‚Äî array of HTML strings for choice buttons (`ABCD`/`multiSelect`). Empty array (`[]`) for `PF` and `open` exercises.
- `correctAnswerIndex` ‚Äî (`ABCD`) 0-based index into `answers` of the correct option (`0` = A, `1` = B, ‚Ä¶). `-1` or a missing field marks an unfilled exercise: clicking an answer just toggles a single `?` feedback marker instead of grading. `loadExercises` `console.warn`s if the index is out of range.
- `correctAnswerIndices` ‚Äî (`multiSelect` only) array of correct indices; its length is the number of picks required. Clicks toggle a light-blue `.selected` state; once the required count is selected the picks are graded (green/red) and partial credit is granted (`maxScore ¬∑ hits / required`, rounded). Deselecting re-opens the choice. When the pick-limit is already reached, clicking a *new* answer replaces the earliest selection (a sliding window ‚Äî the `selected` `Set`'s insertion order makes the oldest pick its first element) instead of doing nothing, so the student never has to manually un-click to change an answer.
- `statements` ‚Äî (`PF` only) array of `{ text, answer }` (`answer: true` = P). Each statement renders as a `.pf-row` with its own P/F buttons graded immediately; the exercise scores `maxScore` only when **all** statements are answered correctly, else 0 (CKE-style).
- `blanks` ‚Äî (`fillIn` only, currently zad 10 and 29) array of `{ label, accepted }`: `label` is the sentence HTML rendered left of the text input, `accepted` the list of recognised answers (compared after `normalizeAnswer` on both sides, so one canonical form like `"(-4,4]"` covers `(‚àí4, 4‚ü©`, `(-4;4>` etc.; list numeric variants like `"6,38"`/`"6.38"` explicitly). Editing an input clears its previous green/red mark; "Sprawd≈∫" grades all blanks and scores `maxScore ¬∑ hits / blanks` (rounded).
- `maxScore` ‚Äî the exercise's point value. `0` marks a "parent" entry (shared intro of a multi-part exercise): no answers UI, no badge (removed from DOM), contributes nothing to the total. All answer paths go through the shared `setScore(points)` helper; `totalScore` is recomputed as the sum of every `earnedScore` (`updateTotalScore()`), so scoring is idempotent and self-correcting ‚Äî no `isScoreGiven` flag or manual add/subtract.
- `selfScore` ‚Äî (`open` only) `true` renders the self-assessment row ("RozwiƒÖ≈º na kartce, por√≥wnaj z rozwiƒÖzaniem i oce≈Ñ siƒô:" + buttons `0 pkt`..`maxScore pkt`) that implements the product's off-platform-solve/self-grade flow.
- `hint` ‚Äî HTML shown/hidden by the "Podpowied≈∫" button; empty string hides the button entirely.
- `formulasPage` ‚Äî page number into `wybrane_wzory_matematyczne.pdf`, or `null` to hide the "Poka≈º potrzebne wzory" button. Clicking it calls `openFormulasAtPage(page)`, which replaces the PDF `<object>` inside `#tablica-wzorow-panel` entirely (changing just `data` doesn't reliably reload PDF viewers) and then shows the panel via `showFormulasPanel()`. Show/hide is centralised in `showFormulasPanel()`/`hideFormulasPanel()` (they toggle the panel's `display` and keep the `#toggle-tablica` label in sync); the bar button and the in-panel close button (`#tablica-close`, the ‚úï in its top-right corner) both go through them. A mirrored second panel `#zasady-oceniania-panel` (left-hand side) shows the official CKE answer/marking PDF whose path comes from `meta.zasadyPdf` (set on the `<object id="zasady-oceniania">` by `applySheetMeta`, resolved through `mediaPath` + `encodeURI`); it has its own `#toggle-zasady` bar button and `showGradingRules()`/`hideGradingRules()`/`#zasady-close`, but no per-page targeting. Both panels are "unlocked": `makePanelDraggable(panel)` makes the top strip (`.panel-uchwyt`) drag the panel (switching it to `left/top` positioning) and the bottom-right grip (`.panel-rozmiar`) resize it, using pointer capture on the handles because the PDF `<object>` swallows mouse events.
- `solutionText` / `solutionTextMore` ‚Äî HTML for the solution panel; `solutionTextMore` is behind an extra "Poka≈º wiƒôcej" toggle and its container auto-hides when empty.
- `solutionStepByStep` ‚Äî either `null` or an array of `{ type: "video"|"image"|"text", src, text }` steps rendered one at a time with prev/next navigation inside the solution panel. **Step switching is double-buffered against flicker:** `showStep()` builds the next step's DOM detached (and KaTeX-renders it), and when the *previous* step is still on screen and the new step has a video, the old step stays visible (frozen on its last frame) until the new video has decoded its first frame (`loadeddata`, `readyState >= 2` shortcut for cached files, 1.5 s `setTimeout` fallback) ‚Äî only then does `replaceChildren` swap the content and `video.play()` start playback (the `autoplay` attribute is stripped so the buffered video doesn't start early). A per-exercise `stepSwapToken` plus a `krokWstawiony` flag guard against rapid ‚óÑ/‚ñ∫ clicking and against the `loadeddata`+timeout pair both firing. The nav (counter, disabled arrows, `markCorrectAnswer` on the last step) updates immediately on click, not at swap time. `renderStep()` still returns a plain HTML string (video steps are wrapped in a `.step-video` div holding the `<video>`, a small `.video-overlay-icon` marker, horizontally centred at the bottom of the film ‚Äî two vertical bars (pause) drawn with `::before`/`::after` when the video is paused mid-play, or a ‚Ü∫ replay arrow once it has ended (driven by `.paused`/`.ended` classes that `showStep` toggles from the video's `paused`/`ended` state; clicking an ended video replays it from the start) and a thin grey `.video-progress`/`.video-progress-bar` (the bar is driven by a `requestAnimationFrame` loop in `showStep` that runs only while the video plays and self-terminates when it pauses/ends or the step's `<video>` leaves the DOM (`!isConnected`); no CSS transition ‚Äî an earlier `timeupdate`+transition combo made the bar move rubber-band-like); the caption goes in a `.step-comment` div); the wiring of the real `<video>` lives in `podepnijSterowanieWideo(video)`, called at swap time: it sets `defaultPlaybackRate`/`playbackRate` (currently `1`), and wires click-to-toggle play/pause, a `.paused` class on `.step-video` (which reveals the ‚ñ∂ icon whenever the video is paused or ended), and the rAF progress-bar loop described above. Doing all this on the attached element is deliberate ‚Äî JS properties/listeners can't be set on the throwaway string `renderStep` returns. Reaching the last step calls `markCorrectAnswer()`, which reveals the correct answer button (guarded against a missing button) ‚Äî i.e. finishing the walkthrough is how open-style exercises "give away" the answer. The "RozwiƒÖzanie" button itself is hidden in `loadExercises` when an exercise has no solution content at all (no text/steps/interactive/more).
- `solutionInteractive` ‚Äî either `null` or the **name** (string) of one of the `widget*` functions described under "Rendering model", e.g. `"widgetOsLiczbowa"`; `loadExercises` resolves it through the `WIDZETY` registry in solutionsInteractive.js and calls the function with the exercise's `.solution-interactive-container`. Nine exercises currently have widgets (1, 5, 9, 10, 12.1, 15, 18, 20, 30).

## Asset folder convention

Every sheet keeps its assets under **`matura/<id>/media/zadN/`** (one `zadN/` folder per exercise number, not array index), e.g. `matura/2024-grudzien/media/zad1/zad1rozw_step1.mp4` (rozwiƒÖzanie = solution), `matura/2024-grudzien/media/zad10/zad10rys.png` (a real figure). In `exercises.json` these are written **relative to the sheet folder** with the `media/` prefix ‚Äî `"media/zad1/zad1rozw_step1.mp4"`, `<img src="media/zad10/zad10rys.png">` ‚Äî the same convention for both sheets.

**Path resolution ‚Äî `mediaPath(src)`.** Because `template.html` renders from the repo root (not from inside the sheet folder), a bare `media/‚Ä¶` path would resolve against the root. `mediaPath(src)` (top of `script.js`) prepends `matura/${SHEET_ID}/` to any non-absolute path (leaves `http(s):`/`data:`/`/‚Ä¶` untouched). It is applied in three places: the `<img>`/`<video>`/`<source>` `src` inside step HTML (`renderStep`), the next-step preload, and ‚Äî via a `querySelectorAll("img[src]")` sweep of the freshly built exercise clone (just before `renderMath`) ‚Äî every `<img>` embedded in the question/hint/solution HTML. The `meta.zasadyPdf` path goes through `mediaPath` too. So **paths in `exercises.json` stay sheet-relative; `mediaPath` is the single place that joins them to the sheet folder** ‚Äî don't hard-code `matura/<id>/` into the data.

Keep filenames **lowercase** ‚Äî `zad17.PNG` once broke on case-sensitive hosting and had to be renamed. Since the KaTeX migration (2026-07-06) the formula crops `zad2/zad2.png`, `zad6/zad6*.png`, `zad7/zad7*.png`, `zad8/zad8.png`, `zad10/zad10.png` and the whole `zad16/` folder were no longer referenced (their content is typeset in KaTeX) and were deleted. The 2026-maj figures (`media/zad12/‚Ä¶` etc.) are still to be cut from that sheet's exam PDF, so those paths currently 404 ‚Äî expected until the images are added.

Each sheet's own exam + answer-key PDFs live in its folder too (`matura/<id>/matematyka-<id>-‚Ä¶-odpowiedzi.pdf`, referenced by `meta.zasadyPdf`; the plain exam PDF is archived alongside). The shared formula sheet `wybrane_wzory_matematyczne.pdf` stays in the repo root. Other/unwired sheets and their text extracts sit in `inne arkusze PDF/`.

Step-by-step solution videos are produced externally with **Manim** (the Python math-animation library), then exported as short `.mp4` files and dropped into the relevant `media/zadN/` folder ‚Äî they aren't generated or editable from within this repo, only referenced and played back (see `solutionStepByStep` above).

## Scoring / UI chrome

`maxTotalScore` is computed inside `loadExercises()` as the sum of all `maxScore` fields (the old hand-maintained constant at the top of the data file is gone ‚Äî nothing to keep in sync). It renders the running total in `#total-score`. `#score-switch-button` (in the `#bar-menu` dropdown) cycles through three display modes (all per-exercise scores / total only / nothing) by toggling `.exercise-score` element visibility, purely client-side, no persistence. `#show-all-solutions` (also in `#bar-menu`) is a toggle that opens/closes every exercise's solution panel by clicking each exercise's own "RozwiƒÖzanie" button (collected in the `wszystkieRozwiazania` array during render), skipping exercises already in the target state; its label flips between "poka≈º/schowaj wszystkie rozwiƒÖzania". `#reset-scores` (third item in `#bar-menu`) clears the saved progress and reloads. Answer progress itself *is* persisted ‚Äî see "Progress persistence" under Rendering model.

## Exam mode (tryb egzaminacyjny)

"Rozpocznij pr√≥bny egzamin" (`#egzamin-start`, in the `#bar-menu` dropdown) runs a timed mock exam. **Design principle: grading and progress-saving keep running through the exact same code path as normal mode** ‚Äî the mode only adds a `body.tryb-egzaminu` class, and style.css under that class hides everything that reveals the result: `.correct`/`.incorrect` are restyled to the neutral blue "selected" look, score badges / `#total-score` / hint / solution / self-score rows / `#toggle-zasady` (the CKE answer-key PDF) / the other `#bar-menu` items are `display: none !important` (important because some of those get inline `display` from JS). The CKE formula sheet (`#toggle-tablica`, per-exercise "Poka≈º potrzebne wzory") deliberately stays available ‚Äî it is allowed on the real matura.

Mechanics (all in the `TRYB EGZAMINACYJNY` section of the inline script): state is `{ start: timestamp }` under its own per-sheet localStorage key `matematykazen-egzamin-<SHEET_ID>` (`KLUCZ_EGZAMINU`, e.g. `‚Ä¶-2024-grudzien`), fully independent of the answer-progress key, so a page refresh resumes the exam (the class is applied at script start, before any rendering ‚Äî no flash of scores). Starting the exam `confirm()`s, wipes the saved progress (an exam starts on a clean sheet) and reloads. `tickExam()` ticks every second into `#egzamin-timer` (tabular-nums; `.malo-czasu` turns it red under 10 min) and auto-ends the exam at 0 ‚Äî guarded on `exercises.length` so that a timer that expired while the tab was closed ends the exam right after the data loads (`startSheet` calls one tick after `loadExercises`). Ending (`#egzamin-koniec` ‚Äî a static button in template.html placed *after* `#exercises-wrapper`, so it sits under the last exercise; `display: none` outside exam mode ‚Äî or timeout) removes the key + class and fills the `#egzamin-podsumowanie` overlay: closed-exercise score (`earnedScore` sum vs non-`selfScore` `maxScore` sum), percentage, time used, and an explicit note that **open (selfScore) exercises are scored by self-assessment only now, after the exam** ‚Äî their rows and the solutions just got unlocked, and self-graded points flow into the same total. Answers given during the exam keep their colours-revealed state after it ends (same DOM, class removed).

## CSS & layout reference

[style.css](style.css) is a single flat stylesheet (no preprocessor, no CSS variables yet). Fonts come from a Google Fonts `@import` at the top (`STIX Two Math` for math, `Lora` fallback). The page is one centred column on a white background; `body` is `text-align: center` and most block widths are fixed at **650px** (`#bar-container` conceptually and `.exercise-container`). Section banners in the file use `/* ===== NAZWA ===== */` comment headers, in Polish ‚Äî follow that convention.

### Top bar (`#top-bar`)

Restructured to a **flexbox three-zone layout** (previously each element used `position: absolute`/`fixed` with hard-coded negative offsets like `right: -330px`, which overlapped whenever the title, score width, or window size changed ‚Äî do **not** reintroduce that pattern).

- `#top-bar` ‚Äî `position: fixed`, full width, bottom border + subtle shadow, `z-index: 10`. This is the only element in the bar that is positioned/fixed.
- `#bar-container` ‚Äî full-width **CSS grid** row (`display: grid; grid-template-columns: 1fr auto 1fr; align-items: center`, padded `12px 30px`, no `max-width` ‚Äî it spans the viewport like the original design so the title can be centred on the whole screen). Holds exactly three zone divs, one per grid column:

### Responsive breakpoints

The layout is still desktop-first (650px column), but a `/* ===== RESPONSYWNO≈öƒÜ ===== */` section at the end of style.css adapts it downwards; **no horizontal scroll from 360px up** (verified by a Playwright sweep at 360/400/720/800/900/1000/1024/1100/1280). The breakpoints:

- `max-width: 1024px` ‚Äî the `.exercise-score` badge moves from `right: -120px` (outside the card; fits only when the window is ‚â• ~1010px) into the card's top-right corner.
- `max-width: 900px` ‚Äî `#exercises-sheet-title` is hidden (it truncated into uselessness around there anyway).
- `max-width: 720px` ‚Äî bar paddings/buttons compact (`#toggle-tablica`/`#toggle-zasady` drop their `min-width: 200px`); `.exercise-container` becomes fluid (`width: auto; max-width: 650px`, smaller padding); step videos scale to container width (`aspect-ratio: 420/180` keeps the shape); answer buttons go to `min-width: 38%` (‚âà2 per row) **except** `.pf-row button`, which must stay 44px (the P/F buttons share `.button-container` with the ABCD ones, so they'd otherwise inherit the widening); PDF panels span 94% of the screen.
- `max-width: 560px` ‚Äî the bar grid collapses to one column and two rows (`#bar-left` on top, `#bar-right` below, both centred; `#bar-center` display:none), `#exercises-wrapper`/`#bar-menu` offsets grow to match the taller bar.

Wide KaTeX display blocks get their own `overflow-x: auto` (on `.katex-display`, in the KaTeX rule near the top of the file) instead of stretching the page.
