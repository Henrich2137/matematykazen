# ARCHITECTURE_CSS.md

CSS & layout reference for this repo — the "how is [style/](style/) built and why" chapter split out of [ARCHITECTURE.md](ARCHITECTURE.md) (see that file for the JS/data architecture). **Keep this file in sync**: whenever you change the styles it describes, update the matching section here in the same edit.

**File split** (no preprocessor, plain CSS in all five): `style/base.css` (fonts, KaTeX sizing, theme/colour variables, `body`, global error message), `style/sheet.css` (top bar, PDF panels, exercise cards, all answer-type styles, solution panel, widgets, misc forms/tables), `style/landing.css` (`.landing-*`, index.html only), `style/exam.css` (exam mode + open-exercise indicators), `style/responsive.css` (breakpoints). **Load order is the cascade and must be preserved**: `base` → `sheet` → `landing` → `exam` → `responsive` (`responsive.css` last, so its overrides win). `template.html` `<link>`s all five; `index.html` only needs `base.css` + `landing.css`. The page is one centred column; `body` is `text-align: center` and most block widths are fixed at **650px** (`#bar-container` conceptually and `.exercise-container`). Section banners use `/* ===== NAZWA ===== */` comment headers, in Polish — follow that convention.

**Theme / colour variables (dark mode).** All colours go through CSS custom properties defined in `:root` near the top of `style/base.css` (semantic tokens: `--bg`, `--bg-muted`, the `--text*` tiers, `--border*`, `--accent-*`, `--correct`/`--incorrect`, `--score-*`, `--overlay-bg`, `--canvas-bg`, …). `:root` holds the **light** values (unchanged from the pre-variable design). The **dark** palette is written in **two twin blocks that must be kept identical**: `@media (prefers-color-scheme: dark) :root:not(.theme-light)` (system/auto) and `html.theme-dark` (manual override, wins over the system). Manual light is `html.theme-light` (it just suppresses the system rule, leaving `:root`). `--canvas-bg` intentionally stays white in dark mode because the interactive widgets draw dark strokes onto the canvas. When adding new colours, use a token (add one if needed) rather than a raw hex, and set both a light (`:root`) and a dark (both dark blocks) value. The toggle is `#theme-toggle` in the `⋯` menu (cycles auto → jasny → ciemny), persisted in localStorage under the **global** key `matematykazen-motyw` (no sheet id — shared across sheets); a tiny inline `<head>` script in template.html **and** index.html applies the saved class to `<html>` before first paint to avoid a theme flash.

## Top bar (`#top-bar`)

Restructured to a **flexbox three-zone layout** (previously each element used `position: absolute`/`fixed` with hard-coded negative offsets like `right: -330px`, which overlapped whenever the title, score width, or window size changed — do **not** reintroduce that pattern).

- `#top-bar` — `position: fixed`, full width, bottom border + subtle shadow, `z-index: 10`. This is the only element in the bar that is positioned/fixed.
- `#bar-container` — full-width **CSS grid** row (`display: grid; grid-template-columns: 1fr auto 1fr; align-items: center`, padded `12px 30px`, no `max-width` — it spans the viewport like the original design so the title can be centred on the whole screen). Holds exactly three zone divs, one per grid column:
  - `#bar-left` (left column, left-aligned) → `#logo` ("Matematyka Zen") only.
  - `#bar-right` (right column, right-aligned, `flex-wrap: nowrap`) → on one line, in DOM order: `#egzamin-timer` and `#egzamin-koniec-bar` (both exam-only — hidden outside `body.tryb-egzaminu`), then `#total-score` (hidden *inside* exam mode) and `#menu-button`. The side zones never wrap vertically — do not put `flex-wrap: wrap` back here. `#menu-button` (the `⋯`) toggles `#bar-menu` — a fixed dropdown just under the bar's right edge (defined right after the bar in the HTML, *outside* `#bar-container`) holding the less-used actions, **including `#toggle-tablica`/`#toggle-zasady`** (the formula-sheet/grading-rules panel toggles moved here unconditionally — same on desktop and mobile, see issues/pasek-przyciski-do-menu.md — because they were the one pair of bar buttons that could overflow on narrow screens; now nothing left in `#bar-left`/`#bar-right` can), plus `#show-all-solutions`, `#score-switch-button` and `#reset-scores` ("resetuj punktację" — wipes the localStorage save and reloads). The menu closes on any click outside it (document-level listener; the ⋯ click itself calls `stopPropagation`).
  - `#bar-center` (middle `auto` column, `justify-self: center`) → `#exercises-sheet-title`. Because the two flanking columns are equal `1fr` tracks, it lands at the **true page centre** regardless of how wide the logo or the right group are (a plain flexed centre column drifts off-centre when the side zones differ in width) — without being taken out of flow, so it can never visually overlap the logo or the controls the way the old `position: absolute` version could on narrow windows. `min-width: 0` + `overflow: hidden` + `text-overflow: ellipsis` (with `white-space: nowrap` kept) is what lets the middle column shrink and truncate the title with an ellipsis if the side zones leave too little room for it, instead of overflowing into them or wrapping onto a second line — wrapping was rejected because it would grow `#top-bar`'s height. `pointer-events: none` is a leftover safety net in case any overlap ever happens anyway.
- To add another control to the bar, drop it into the appropriate zone div (or into `#bar-menu` if it's a rarely-used action) — no offsets to recompute. Keep these IDs stable when restyling — they are all looked up by the `app/*.js` code (`getElementById`), so deleting or renaming one silently breaks a feature even though the markup "looks" inert:
  - **Bar chrome:** `#logo`, `#total-score`, `#menu-button`, `#bar-menu`. Exam-only bar controls (hidden outside `body.tryb-egzaminu`, but always in the DOM): `#egzamin-timer`, `#egzamin-koniec-bar`. Mode line in `#bar-center`: `#exercises-mode-subtitle`.
  - **`#bar-menu` items** (all `getElementById`-wired): `#toggle-tablica`, `#toggle-zasady`, `#theme-toggle`, `#show-all-solutions`, `#score-switch-button`, `#reset-scores`, `#egzamin-start`, `#zegar-toggle`, `#wskazniki-tryb-toggle`, `#natychmiastowa-toggle`, `#sprawdz-wszystkie`, `#zglos-blad-toggle`.
  - **PDF panels:** `#tablica-wzorow-panel`, `#tablica-wzorow`, `#tablica-close`, `#zasady-oceniania-panel`, `#zasady-oceniania`, `#zasady-close`.

  The `#bar-left/center/right` wrappers are layout-only and JS-free. `#toggle-tablica`/`#toggle-zasady` live in the DOM inside `#bar-menu` now, but `app/panels.js` only ever looks them up by `getElementById`, so the move needed no JS changes.
- `#tablica-wzorow-panel` — the floating formula-sheet panel: `position: fixed`, top-right, `display: none` until the JS toggles it to `block`. `z-index: 9` (just under the bar). It's a container around the PDF `<object id="tablica-wzorow">` (which fills it at `width/height: 100%`, borderless) plus `#tablica-close` — the ✕ close button absolutely positioned in the panel's top-right corner (deliberately large: 40px, bold ✕, border + shadow, so it doesn't vanish against the PDF). The panel owns the border/shadow/size; the `<object>` is what `openFormulasAtPage` swaps out, so keeping the chrome on the wrapper means a page change never loses the border or the close button. `#zasady-oceniania-panel` shares all these rules (selector lists) and only overrides `right: auto; left: 30px`. Both carry class `.pdf-panel` (`box-sizing: border-box; padding-top: 46px` making room for the drag strip): `.panel-uchwyt` (46px top strip, `cursor: move`, `touch-action: none`) and `.panel-rozmiar` (22px bottom-right grip, `cursor: nwse-resize`) are the drag/resize handles wired by `makePanelDraggable`.

## Exercise cards & solution panel

- `.exercise-container` — the per-exercise row (650px wide, centred with `margin: auto`, `padding: 50px`, left-aligned text, `font-size: 18px`), cloned from the hidden `#exercise-template`. Exercises are **not** boxed cards any more — instead each row is separated from the next by a single hairline `border-bottom: 1px solid #f0f0f0` (the old full `border` + `margin: 20px auto` card look is kept commented out just above, in case of a revert). `position: relative` so `.exercise-score` (the per-exercise points badge) can sit at its top-right; the badge colour tier is one of `.low-/.mid-/.high-exercise-score`. Figure images in the question (`.question img`) are forced to `display: block; margin: 15px auto` so they sit centred on their own line instead of floating mid-sentence and shifted right; a genuinely inline image (e.g. Zad 2's number inside "Liczba … jest równa") must opt out with an inline `display:inline-block; vertical-align:middle` in its own `style`.
- `.button-container button` — multiple-choice answer buttons (inline-block, `1px solid #e7e7e7` border). Sizing is `min-width: 22%` + `white-space: nowrap` (not a fixed `width`): short numeric answers keep an even grid, while a longer KaTeX formula (interval, fraction) widens its own button instead of wrapping the "A." onto a separate line above the math. `vertical-align: middle` on the button (and on any `img` inside it) keeps the option letter (A., B., …) centred on the vertical axis rather than sitting at the bottom next to a taller neighbour. State classes toggled by JS: `.correct` (green border) / `.incorrect` (red border) / `.selected`. All marked states get a visually 2× thicker border via `box-shadow: inset 0 0 0 1px <border colour>` — a real thicker `border` would shift the layout (same trick on `.pf-row`, `.self-score-container` and `.fill-in-input` states).
- `.answer-check-floating` — the per-exercise "sprawdź" button shown only in deferred-grading mode (see ARCHITECTURE.md "Immediate vs. deferred grading"). `.answers-container` is `position: relative` so this button is `position: absolute; left: calc(100% + 12px); top: 0` — floated just right of the answers frame, out of flow (never changes the exercise's height), left of the indicator dot column. Its selector `.button-container .answer-check-floating` deliberately outranks `.button-container button` (width/margins/font reset). Below `900px` it falls back to in-flow (`position: static`, centred under the answers) to avoid overflow — the only place it affects height. Hidden in exam mode.
- `.light-button` / `.light-button-container` — the borderless text actions ("Podpowiedź / Rozwiązanie / Pokaż potrzebne wzory", step arrows, "Pokaż więcej").
- `.hint-container` / `.solution-container` — both `display: none` by default, revealed by JS. Inside the solution panel, the sub-blocks `.solution-text-container`, `.solution-step-by-step-container`, `.solution-interactive-container` each use `:empty { display: none }` so they self-hide when the exercise has no such content. Step media sizing lives in `.solution-step-by-step-container video/img`; each step's caption is wrapped by `renderStep()` in a `.step-comment` div with a ~2-line `min-height`, and `.steps-content` has `min-height: 250px`, so the step-by-step panel keeps a roughly constant height (the nav doesn't jump) until a caption runs past ~2 lines. An empty `.step-comment` shows a subtle "· · ·" filler via `:empty::before`. The blocks are separated by `border-bottom` dividers; `loadExercises` strips the divider off the **last visible** block (computed from `hasText`/`hasSteps`/`hasInteractive`/`hasMore`) so a text-only solution has no dangling separator underneath. The revealed "Pokaż więcej" content (`.solution-text-more`) is padded `0 40px` so it isn't glued to the left edge.

## Answer-type & widget styles

- `.pf-row` — a P/F statement row: flex, statement text (`.pf-text`, `flex: 1`) + fixed-width (44px) P and F buttons; same green/red `.correct`/`.incorrect` border states as the choice buttons.
- `.multi-select-container button` — `min-width: 30%` (long formulas need more room than the 22% ABCD buttons; same grow-don't-wrap rule); the pre-grading toggle state is `.selected` (light-blue border, also defined for `.button-container button.selected`). `.multi-select-note` is the small grey "Wybierz dwie odpowiedzi." caption.
- `.self-score-container` — the self-assessment row for open exercises: grey label + pill buttons; the chosen one gets `.selected` (light-blue border, pale blue background).
- `.fill-in-container` — the `fillIn` answer area: `.fill-in-row` flex rows (`.fill-in-label` sentence + `.fill-in-input` text field with `.correct`/`.incorrect` states), plus the centred `.fill-in-check` "Sprawdź" button. The dedicated input styling matters — the global `input { width: 24px }` rule would otherwise shrink the text fields to nothing.
- `.widget` — interactive-widget wrapper inside `.solution-interactive-container`: `.widget-title` (grey intro line), `.widget-canvas` (`max-width: 100%`, `touch-action: none` so touch-dragging doesn't scroll the page), `.widget-controls`, `.widget-readout` (fixed `min-height` sized for **two lines** — values line + `<br>` + the ✓/✗ status comment, which every widget puts on its own second line — so messages don't make the widget jump). Range inputs inside `.widget` are widened to 320px (the global `input { width: 24px }` rule is meant for the tiny number inputs). Status colours: `.wg-ok` green, `.wg-zle` red, `.wg-neutral` grey; `.widget-formula-list`/`.wg-formula(.active)` is the highlighted piecewise-formula list used by the zad 10 widget. Zad 1 extras: `.wg-a`/`.wg-b` colour the a/b number inputs (green/blue) to match the centre point and distance arrows on the axis, `.wg-reset` is the compact reset button. The old `.mathText` helper class is gone — all widget math is rendered by `wgMath` (KaTeX), see "Rendering model" in ARCHITECTURE.md.
- `::selection` — light-blue background, text colour unchanged (deliberate: the default white-on-navy looked off against the minimalist page).
- `.data-table` — bordered, centred data tables embedded in question HTML (zad 29); header cells grey-tinted, normal font weight.
- **Zgłaszanie błędów** (bottom of `style/sheet.css`, `/* ===== ZGŁASZANIE BŁĘDÓW W ZADANIU ===== */`, all colours via base.css tokens so dark mode is automatic): `.report-error-link` is the discreet per-exercise text button (borderless, `--text-faint`, underline on hover), hidden wholesale by `body.bez-zglaszania .report-error-link { display: none }` when the `#zglos-blad-toggle` menu option is off (`#zglos-blad-toggle` itself is added to the bordered menu-button selector list so it matches the other menu toggles). The modal `#zglos-blad-overlay` / `.zglos-blad-okno` mirrors `#egzamin-podsumowanie` (fixed full-screen `--overlay-bg`, centred `--bg` window with `--border-window`) but at a higher `z-index` (21, above the exam summary's 20). Form fields (`.zglos-blad-okno textarea`/`input[type="email"]`) must re-declare `width: 100%` **and** `text-align: left` to escape the global `input { width: 24px; text-align: center }` rule (same gotcha as `.fill-in-input`). The honeypot wrapper `.zglos-blad-hp` is pushed off-screen (`position: absolute; left: -9999px`) rather than `display:none` so bots still fill it. The `.zglos-toast` (bottom-centre, fade via `opacity`/`transform`, `z-index: 22`) uses a `--correct` border by default and `--incorrect` under `.zglos-toast-blad`.
- `.landing-*` — the whole landing page (index.html): `body.landing-page` is a 700px column; `.landing-hero` (h1 + `.landing-lede` + `.landing-cta` bordered button that inverts on hover), `.landing-cards` (3 flex cards, stack under 640px via the file's only media query), `.landing-section-alt` disclaimer, `.landing-footer`. `#logo` in the exam sheet is an `<a>` styled to look like the old plain text (black, underline on hover).

## Responsive breakpoints

The layout is still desktop-first (650px column), but `style/responsive.css` (the whole file, one `/* ===== RESPONSYWNOŚĆ ===== */` section, loaded last so it wins the cascade) adapts it downwards; **no horizontal scroll from 360px up** (verified by a Playwright sweep at 360/400/720/800/900/1000/1024/1100/1280). The breakpoints:

- `max-width: 1024px` — the `.exercise-score` badge moves from `right: -120px` (outside the card; fits only when the window is ≥ ~1010px) into the card's top-right corner.
- `max-width: 900px` — `#exercises-sheet-title` is hidden (it truncated into uselessness around there anyway).
- `max-width: 720px` — bar paddings/buttons compact; `.exercise-container` becomes fluid (`width: auto; max-width: 650px`, smaller padding); step videos scale to container width (`aspect-ratio: 420/180` keeps the shape); answer buttons go to `min-width: 38%` (≈2 per row) **except** `.pf-row button`, which must stay 44px (the P/F buttons share `.button-container` with the ABCD ones, so they'd otherwise inherit the widening); PDF panels span 94% of the screen.
- `max-width: 560px` — the bar grid collapses to one column and two rows (`#bar-left` on top, `#bar-right` below, both centred; `#bar-center` display:none), `#exercises-wrapper`/`#bar-menu` offsets grow to match the taller bar.

Wide KaTeX display blocks get their own `overflow-x: auto` (on `.katex-display`, in the KaTeX rule near the top of `style/base.css`) instead of stretching the page.

## Known layout gotchas

- Spacing below the fixed `#top-bar` is `padding-top: 100px` on `#exercises-wrapper` (the old `<br><br><br><br>` spacer is gone). If you change the bar height noticeably, adjust that padding **and** the 560px-breakpoint override (110px).
